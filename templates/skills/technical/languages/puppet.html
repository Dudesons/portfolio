Puppet


Puppet is a configuration management system that allows you to define the state of your IT infrastructure, then automatically enforces the correct state.
Whether you're managing just a few servers or thousands of physical and virtual machines, Puppet automates tasks that sysadmins often do manually, freeing up time and mental space so sysadmins can work on the projects that deliver greater business value.
Whether you're deploying vendor-supplied applications or working with a team of internal software developers, Puppet automates every step of the software delivery process: from provisioning of physical and virtual machines to orchestration and reporting; from early-stage code development through testing, production release and updates.
Puppet ensures consistency, reliability and stability.
It also facilitates closer collaboration between sysadmins and developers, enabling more efficient delivery of cleaner, better-designed code.

Puppet point of view

Puppet begins to gain interest from the time when we start having to manage at least 10 machines. It allows to set up an architecture for managing the IT infrastructure as highly accomplished like :
  • Roles and profiles management
  • Outsourcing data so that different nodes may have information between them
  • The use of clear language
  • Management of extensive machine feature (physical machine or virtual, what hypervisor type, OS type, Os version, CPU usage ...)
  • An open source version and an enterprise
  • A strong community around puppet
  • A set of tools enhancing Puppet like puppetlint to verify if the "style guide" of puppetlabs is respected, the RPSEC that allows to make TDD (Test Driven Document), Beeker (test aceptance) R10K (deployment of puppet modules) ...

Compared with its concucents. Puppet has three main competitors which are Chef SaltStack & Ansible. Chef lot like puppet but I could not work on it to make me a sufficiently constructive notice
Saltstack & Ansible differs from puppet that their system is based on yaml files.
Saltstack uses Jinja template also to enable the implementation of conditional system. It is based on a master slave model as a puppet but a connection stability issues on cloud or between cloud environements is unstable.
Ansible has a system conditional programming but in yaml syntaxe like, it also differs by having a single machine and then control the infrastructure nodes via ssh.

Point de vue de Puppet

Puppet commence à prendre de l'interêt à partir du moment où on commence à devoir gérer au moins 10 machines. Il permet de mettre en place une architecture de la gestion du parc informatique très abouti comme :
  • la gestion de rôles et de profiles
  • l'externalisation des données afin que différents noeuds peuvent avoir des information entre eux
  • L'utilisation d'un langage clair
  • Une gestion des caractéristique des machines poussées (machine physique ou virtuelle, quelle type d'hyperviseur, type d'OS, version de l'OS, utilisation du processeur ...)
  • Une version opensource et une "enterprise"
  • Une forte communauté autour de puppet
  • Un ensemble d'outils renforçant Puppet, comme les puppetlint qui permet de vérifier si le "style guide" de puppetlabs est respecté, les rpsec qui permette de faire du TDD (Test Driven Document), Beeker (test d'aceptance), R10K (déploiement des modules puppet) ...

Comparé à ses concucents. Puppet a trois principale concurents qui sont Chef, SaltStack & Ansible. Chef ressemble beaucoup à puppet mais je n'ai pas pu travailler dessus pour me faire un avis suffisament constructif
Saltstack & Ansible se différencie de puppet par le fait que leur système est basé sur des fichiers yaml.
Saltstack utilise également le système de template jinja pour permettre la mise en place de système de conditions. Il est basé sur un modèle master slave comme puppet mais a des problèmes de stabilité de connexion sur des environements de cloud ou inter cloud.
Ansible a un système de conditions aussi mais sous forme de yaml, il se différencie aussi par le fait d'avoir qu'une seule machine puis de controler les noeuds de l'infrastructure par ssh.
