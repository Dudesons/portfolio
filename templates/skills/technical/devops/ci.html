Continuous Integration



            “Continuous Integration doesn’t get rid of bugs, but it does make them dramatically easier to find and remove.”
— Martin Fowler, Chief Scientist, ThoughtWorks

Solve problems quickly

Two goals of the continuous integration is :
  • The software is in a working state all the time.
  • To determine if the current release candidate is not the release

Because you’re integrating so frequently, there is significantly less back-tracking to discover where things went wrong, so you can spend more time building features.
Continuous Integration is cheap. Not continuously integrating is costly.
If you don’t follow a continuous approach, you’ll have longer periods between integrations.
This makes it exponentially more difficult to find and fix problems.
Such integration problems can easily knock a project off-schedule, or cause it to fail altogether.
Continuous integration is often based on the implementation of a software component to automate tasks like : compilation, unit and functional testing, product validation, performance testing...
At each change of the code, the software component will perform a set of tasks and produce a set of results (test, code coverage ...),
after the developer can then consult it. This integration lets not forget items at the start of production and therefore improve the quality of the product.

To apply this technique, you must first:
  • The source code is shared (using version control software such as git, svn, etc.)
  • Developers integrate (commit) their changes daily
  • Integration tests are developed to validate the application


A continuous integration server is then required, as Jenkins.Other tools, like SonarQube can be set up a monitoring of the quality of the code.
The main advantages of this technique are developing :
  • The immediate test changes
  • Early notification in case of inconsistent or missing code
  • Integration problems are detected and repaired continuously, avoiding the problems of last minute
  • A version is always available for testing, demonstration or distribution

Continuous Integration



Deus principaux ovjectifs du Continuous Integration sont :
  • Le logiciel est dans un état fonctionnel tout le temps
  • Déterminer si la release candidate actuel n'est pas une release
L'intégration continue repose souvent sur la mise en place d'une brique logicielle permettant l'automatisation de tâches : compilation, tests unitaires et fonctionnels, validation produit, tests de performances...
À chaque changement du code, cette brique logicielle va exécuter un ensemble de tâches et produire un ensemble de résultats (test, code coverage ...), que le développeur peut par la suite consulter.
Cette intégration permet ainsi de ne pas oublier d'éléments lors de la mise en production et donc ainsi améliorer la qualité du produit.

Pour appliquer cette technique, il faut d'abord que :
  • Le code source soit partagé (en utilisant des logiciels de gestion de versions tels que CVS, Subversion, git, Mercurial, etc)
  • Les développeurs intègrent (commit) quotidiennement (au moins) leurs modifications
  • Des tests d'intégration soient développés pour valider l'application


Un serveur d'intégration continue est ensuite nécessaire, tel que Jenkins. D'autres outils, comme SonarQube peut être mis en place afin de superviser la qualité du code.
Les principaux avantages d'une telle technique de développement sont :
  • Le test immédiat des modifications
  • La notification rapide en cas de code incompatible ou manquant
  • Les problèmes d'intégration sont détectés et réparés de façon continue, évitant les problèmes de dernière minute
  • Une version est toujours disponible pour un test, une démonstration ou une distribution

  • Ecriture de tâches pour automatiser les tests et les builds
  • Personnalisation d'un serveur de Continuous Integration en y ajoutant des briques logiciel développées
