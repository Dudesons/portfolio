Continuous Delivery


    “The essence of my philosophy to software delivery is to build software so that it is always in a state where it could be put into production.
    We call this Continuous Delivery because we are continuously running a deployment pipeline that tests if this software is in a state to be delivered.”
— Jez Humble

Getting software released to users is often a painful, risky, and time-consuming process.
Continuous Delivery can help organizations become lean, agile and innovative.
Through reliable, low-risk releases, Continuous Delivery makes it possible to continuously adapt software in line with user feedback, shifts in the market and changes to business strategy.
Test, support, development and operations work together as one delivery team to automate and streamline the build, test and release process.
You’re doing continuous delivery when :
  • Your software is deployable throughout its lifecycle
  • Your team prioritizes keeping the software deployable over working on new features
  • Anybody can get fast, automated feedback on the production readiness of their systems any time somebody makes a change to them
  • You can perform push-button deployments of any version of the software to any environment on demand
You achieve continuous delivery by continuously integrating the software done by the development team, building executables, and running automated tests on those executables to detect problems.
Furthermore you push the executables into increasingly production-like environments to ensure the software will work in production.
To do this you use a Deployment Pipeline.
The principal benefits of continuous delivery are:
  • Reduced Deployment Risk: since you are deploying smaller changes, there's less to go wrong and it's easier to fix should a problem appear.
  • Believable Progress: many folks track progress by tracking work done. If "done" means "developers declare it to be done" that's much less believable than if it's deployed into a production (or production-like) environment.
  • User Feedback: the biggest risk to any software effort is that you end up building something that isn't useful. The earlier and more frequently you get working software in front of real users, the quicker you get feedback to find out how valuable it really is
Obtenir une release d'un logiciel est souvent un processus risqué, douloureux et qui prend beaucoup de temps.
Continuous Delivery peut aider les organisations à devenir plus lean, agile et innovatrice.
Grâce à des release fiables et peu risqué ceci permet de rendre possible d'adapter le logiciel en permanence aux besoins des utilisateurs et d'être très proche de ce qu'ils veulent.
Les équipes de test, support, développement et production travaillent ensemble comme une équipe de delivery pour automatiser et construire les workflows de test, build et de release.
Le but va être de casser les "silos" mise en place dans une entreprise afin que tout le monde travail pour un objectif livrer le logiciel rapidement et bien.
Egalement un autre but est de pouvoir déployer une release logiciel sur n'importe quel environnement et aussi peut importe la version du logiciel en un seul clique.
Les principaux bénéfices :
  • Réduire les risques de déploiement : puisqu'on déploit souvent on déploit des petits changements, il ya moins de risque de se tromper et il est plus facile de fixer si un problème qui apparaît.
  • Progrès crédible: beaucoup de personnes suivent le progrès par si une tâches a été finit. Si une tâche "finit" singifie que "le développeur déclare que c'est fini", c'est beaucoup moins crédible que si on dit que la tâche est fini lors du déploiement en production.
  • Le retour d'utilisateur: Le plus grand risque pour tout effort dans l'édition logiciel est que vous finissez quelque chose qui est inutile. Plus c'est le feedback demandé est fait tôt et fréquemment, le travail effectué sera au plus proche des besoins utilisateurs.
