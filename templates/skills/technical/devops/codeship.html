Codeship


Codeship is a proprietary continuous integration tool only available as a the format Service as a Software.
Codeship is avaible with free plan with limited build and no concurented build.
It supports only two integration of SCM tools (Git and bitbucket).
This is a CI for python, ruby, node.js, php, java and golang technologies
A build can be started by 2 means :
  • Triggered by commit in a version control system
  • Manually

With Codeship, I only set up continuous integration workflow for python.
Also I use bash script for codeship workflow to have no code in the CI and you can easily migrate to another CI if necessary.

Jenkins point of view

Codeship makes it possible to implement quickly a continuous integration as a service.
There is no administrative costs but if you need a package who is not available in the service you have to request it to the support and waiting if they will integrate this.
It's not so flexible and not allowing suddenly able to really personalize their continuous integration system than jenkins.
There is a strong support who gives usually a fast response (except for package integration application).
Codeship is limited by technologies for the CI and it's not possible to install in your personal infrastrucutre.
Codeship leaves the scope of continuous integration because after the CI tasks it deploys directly into the production server instead of an artifact repository. So it mixes the continuous integration and continuous delivery principles

Compared to its concucents, Codeship has the advantage of being usable very quickly, it doesn't need administrator and it have a free plan.
On the other hand it's limited by the scm tools, the technologies languages are limited and it can be run correctly for simple workflow.

Point de vue de Codeship

Codeship permet de mettre en œuvre rapidement un "continuous integration as a service".
Il n'y a pas les coûts d'administration, mais si vous avez besoin d'un paquet qui n'est pas disponibles dans le service que vous devez faire une requête d'intégration du paquet au support et attendre une réponse positive ou négative puis son intégration.
Il n'est pas si "flexible" et ne permet pas vraiment de personnaliser le système de CI comme Jenkins .
Il y a un support technique de qualité qui donne généralement une réponse rapide (sauf pour l'intégration d'un paquet).
Codeship est limité par les technologies de la CI et il est impossible de l'installer dans votre infrastrucutre personnelle.
Codeship sort du scope de la CI parce qu'après les tâches de CI, elle déploie directement vers la production au lieu d'un repository d'artifact. Donc il mélange les principes de continuous integration et de continuous devlivery

Comparer à ses concucents, Codeship a l'avantage d'être utilisable très rapidement, il n'a pas besoin administrateur et a un plan gratuit.
D'autre part il est limité par les outils de SCM, les technologies sont limitées et il fonctionne correctement pour workflow simple.
